## 4. The HydraDB Unlock

Most teams will use HydraDB as "vector DB with auto-extracted entities." That's the floor.

**Embodipedia uses seven HydraDB capabilities, six of which most teams will not touch:**

| Capability | What it enables in Embodipedia |
|---|---|
| **Sub-tenants** | Parallel perspective namespaces (canonical / bull / bear). Same evidence, three interpretations. |
| **Hive Memories** | Cross-agent learning. The extraction agent that processed 50 Brett Adcock tweets writes a "discount timeline claims by 0.7x" lesson; the next extraction reads it. |
| **Function routing** | Self-healing wiki. Stale entities trigger registered refresh functions retrieved as knowledge objects. |
| **`entity_paths`** | Multi-hop relationship chains pre-computed at retrieval time. Powers "How is Pi connected to Karol Hausman" queries and "What links here" pages. |
| **Memory metadata filtering** | Time-travel slider (filter by `published_at`), perspective tabs (filter by `perspective`), source-type breakdowns. |
| **Context graph auto-extraction** | Disambiguation pages ("Pi" the company vs π the constant), entity merging across heterogeneous sources. |
| **Confidence + recall ranking** | Inline `[citation needed]` tags on prose claims that lack primary sources. |

**The defensibility question to anticipate from judges:** "Could you build this on Pinecone or Weaviate?"

Your answer (rehearse this):

> No. The dual-narrative wiki requires perspective-isolated graph subgraphs over a shared evidence pool — HydraDB sub-tenants give us that natively. The self-healing wiki requires callable functions as first-class graph entities — HydraDB's function routing is the only place that exists. The cross-agent learning requires a shared memory tier that's neither user memory nor knowledge — HydraDB's hive memories are the only implementation. Without HydraDB, this is 4 weeks of infrastructure, not 48 hours.

### The five sub-tenants

| Sub-tenant | Purpose | What lives here |
|---|---|---|
| `canonical` | Evidence-grounded ground truth | Claims tagged neutral/primary; the default lens |
| `bull` | Optimistic narrative | Same claims, weighted toward signals supporting "deployments are real, timelines will hit" |
| `bear` | Skeptical narrative | Same claims, weighted toward "demos are staged, valuations are vapor" |
| `articles` | Synthesized wiki prose | Rendered article bodies, infoboxes, Talk page sections |
| `agents` | Hive memories | Lessons agents write to each other (e.g., "TechCrunch tends to amplify funding claims; discount by 0.6") |

**Why three perspectives, not two:** Two creates false binary; three gives the user (and the synthesis agent) a neutral evidence base to anchor reconciliation. The canonical sub-tenant is not the average — it's the evidence-grounded subset that *both* narratives agree on.

## 6. Wikipedia Primitives → HydraDB Mapping

Twelve Wikipedia editorial primitives, each backed by a real HydraDB capability:

| # | Wikipedia primitive | Description | HydraDB primitive used |
|---|---|---|---|
| 1 | **Article page** | The canonical encyclopedia entry | Synthesized memory in `articles` sub-tenant; cited from `canonical` |
| 2 | **Talk page** | Meta-discussion: contested claims, deletion debates, source disputes | Hive memories where agents argue; rendered as Wikipedia Talk markup |
| 3 | **Edit history** | Every change ever made, with diffs | Temporal metadata (`created_at`, `superseded_by`) on every memory |
| 4 | **References section** | Numbered citations at bottom | Source memories linked to claim memories via `subject_entity` + `source_url` |
| 5 | **[citation needed]** | Inline markers on unsupported claims | Confidence threshold check at synthesis time |
| 6 | **Infobox** | Structured side-card with key facts | Lexical recall on canonical structured fields (founded, CEO, valuation, height) |
| 7 | **Categories** | Multi-membership hierarchies at page bottom | `entity_type` metadata; auto-extracted category memberships from context graph |
| 8 | **What links here** | Reverse traversal: who references this page | Reverse `entity_paths` query |
| 9 | **Quality tiers** | "Stub" / "Good Article" / "Featured" stars | Evidence density score per entity (count of primary sources, source diversity) |
| 10 | **Disambiguation pages** | "Pi (planet) vs Pi (company) vs π (math)" | HydraDB's auto entity disambiguation surfaced as UI |
| 11 | **Recent Changes** | Global feed of every edit | Real-time query on `ingested_at` metadata, descending |
| 12 | **Watchlist** | User-specific list of pages to follow | User memories per sub-tenant (`memory_recall`) |

**Why this mapping is the secret weapon:** Most teams will give judges a "wiki UI" that's actually just a fancy chat interface. Embodipedia gives them an actual Wikipedia with all the primitives they've grown up with. The familiarity creates instant credibility, and every primitive has a real HydraDB feature behind it.

- **Platform** — Figure 02, Optimus Gen 2, Neo, Unitree G1, Atlas, Apollo, Phoenix, Digit, π0
- **Person** — Brett Adcock, Bernt Bornich, Aaron Saunders, Karol Hausman, Sergey Levine, Pieter Abbeel, Elon Musk, Marc Raibert, Geordie Rose, Damion Shelton
- **Lab** — Physical Intelligence Lab, NVIDIA Robotics, Toyota Research Institute, Google DeepMind Robotics, Meta AI
- **Capability** — locomotion, manipulation, dual-arm coordination, factory deployment, household tasks, teleoperation
- **Benchmark** — RoboArena, LIBERO, BEHAVIOR, RoboCasa, RT-X, Open-X-Embodiment
- **Foundation Model** — GR00T N1, π0, π0.5, RT-2, OpenVLA, Octo, Helix
- **Hardware** — NVIDIA Jetson Orin, GR00T, Unitree actuators, Tesla Dojo
- **Event** — Tesla AI Day 2024, Figure Series B, Apptronik Apollo launch
- **Claim** — "Figure deployed at BMW," "Tesla shipping 1M units by 2030," "Pi achieves zero-shot manipulation"

### Memory metadata schema (every knowledge memory you write)

```python
{
  # Identity
  "memory_id": "claim_figure_bmw_deployment_2024_10_15_001",
  "entity_type": "claim",              # claim | source | event | benchmark_result
  "subject_entity": "Figure 02",        # what this is about
  "actor_entity": "Brett Adcock",       # who made the claim (null for objective)

  # Provenance
  "source_url": "https://twitter.com/adcock_brett/status/...",
  "source_type": "tweet",              # tweet | paper | blog | podcast | news | filing | demo_video | dev_report
  "evidence_strength": "primary",       # primary | secondary | tertiary
  "source_publisher": "Brett Adcock",   # for attribution

  # Temporality
  "published_at": "2024-10-15T14:32:00Z",
  "ingested_at": "2026-05-16T09:00:00Z",
  "superseded_by": null,                # memory_id of newer claim that overrides
  "supersedes": null,                   # memory_id of older claim this overrides
  "stale_at": "2026-08-15T00:00:00Z",   # when this should be re-verified

  # Perspective (the key field for sub-tenant routing)
  "perspective": "neutral",             # bull | bear | neutral
  "confidence": 0.72,                   # extraction confidence (0..1)

  # Claim semantics
  "claim_type": "deployment",           # capability | timeline | metric | quote | benchmark_result | valuation
  "claim_polarity": "positive",         # positive | negative | uncertain
  "claim_text": "Figure 02 has been deployed at BMW since October 2024",

  # Quality
  "verification_status": "unverified",  # verified | unverified | disputed | refuted
  "verified_by_source_id": null,
}
```

### Sub-tenant routing rules

When you ingest a piece of evidence, it goes to **one or more** sub-tenants based on its perspective tag:

- `perspective: "neutral"` → write to `canonical` only
- `perspective: "bull"` → write to `bull` (and optionally `canonical` if also evidence-grounded)
- `perspective: "bear"` → write to `bear` (and optionally `canonical` if also evidence-grounded)

The same source can produce multiple claim memories with different perspectives. A Brett Adcock tweet making three claims might generate one neutral memory (the verifiable fact: "Figure raised $675M") and one bull memory (the optimistic spin: "this funding round validates our deployment thesis").

## 8. The Agent Pipeline

Five stages. Each stage is a discrete agent (or pool of agents) with clear inputs, outputs, and HydraDB interactions.

### Stage 1: Sourcing Agent

**Role:** Decides what to ingest next.

**Inputs:**
- Seed list of company names, founder Twitter handles, conferences, paper venues
- Hive memory of "what sources tend to be high-signal"

**Outputs:**
- Queue of URLs to ingest, prioritized

**HydraDB calls:**
- `memory_recall` on `agents` sub-tenant: "What sources have been most useful?"
- Writes new lessons back as hive memories after each ingest pass

**Implementation note:** For the hackathon, hardcode a seed list of ~120 URLs. The sourcing agent becomes a curator + scheduler. Don't try to make it autonomous over the open web in 48 hours.

### Stage 2: Extraction Agent Pool (four specialized agents)

Each source type gets its own extractor because extraction strategies differ dramatically.

#### 2a. `tweet_agent`

**Role:** Extracts claims from a single tweet or thread.

**Behavior:**
1. Read hive memory: lessons about this author's reliability
2. Use LLM to extract: claims, subject entities, actor, polarity, confidence
3. Classify each claim's perspective (bull/bear/neutral)
4. Write one knowledge memory per claim to the appropriate sub-tenant(s)
5. Append lesson to hive memory if the author exhibited a pattern (e.g., "Adcock posts deployment claims in October that don't materialize until April")

**HydraDB calls:**
- `memory_recall` from `agents` sub-tenant for author lessons
- `add_knowledge_memories` to `canonical` / `bull` / `bear` as appropriate
- `add_hive_memories` for new author/source lessons

#### 2b. `paper_agent`

**Role:** Extracts structured claims from a research paper.

**Behavior:**
1. Parse paper (use PyMuPDF or arxiv API)
2. Extract: title, authors, affiliations, methods, key claims, benchmarks, comparisons to prior work
3. Each benchmark result becomes a claim with `claim_type: "benchmark_result"` and `evidence_strength: "primary"`
4. Each comparison ("we outperform X by Y%") becomes a claim that may contradict an earlier claim
5. Citations in the paper become source memories

#### 2c. `transcript_agent`

**Role:** Extracts attributed claims from podcast or video transcripts.

**Behavior:**
1. Segment transcript by speaker (use diarization if available, else infer from text)
2. For each speaker turn, extract claims and attribute to that speaker
3. The `actor_entity` field is critical here — distinguishes "Pieter Abbeel said X" from "the host said Pieter Abbeel said X"

#### 2d. `news_agent`

**Role:** Extracts claims from journalism (TechCrunch, Reuters, The Information, etc.)

**Behavior:**
1. Read article
2. Distinguish reported facts from quoted claims (this is critical for accuracy)
3. Reported facts → `evidence_strength: "secondary"` (journalist's claim)
4. Quoted claims → `evidence_strength: "primary"`, with `actor_entity` set to the quoted person

### Stage 3: Synthesis Agent (per entity, async)

**Role:** Compiles wiki articles from extracted claims.

**Behavior:**
1. For target entity E, query all three perspective sub-tenants:
   ```python
   canonical = hydradb.full_recall(sub_tenant="canonical",
                                    query=f"everything about {E}",
                                    metadata_filter={"subject_entity": E})
   bull = hydradb.full_recall(sub_tenant="bull", ...)
   bear = hydradb.full_recall(sub_tenant="bear", ...)
   ```
2. Construct an article in three sections:
   - **Article body:** Synthesized from `canonical` results. Every prose claim must trace to ≥1 source.
   - **Infobox:** Lexical recall on structured fields.
   - **Talk page sections:** Where `bull` and `bear` disagree, render their arguments side-by-side. Where neither resolves, add an "Open Question" entry.
3. For every prose sentence, check confidence:
   - Confidence ≥ 0.7 + ≥ 1 primary source → render with citation footnote
   - Confidence < 0.7 OR only secondary sources → add `[citation needed]` tag
4. Write synthesized article to `articles` sub-tenant with metadata pointing back to source claims.

**Key design choice:** The synthesis agent does NOT call the LLM to "write a Wikipedia article about Figure." That would produce hallucination-prone prose. Instead, it queries HydraDB for evidence chunks, then asks the LLM to *organize* those chunks into article structure — with strict instructions to use only the provided evidence and to tag uncertain claims.

### Stage 4: Contradiction & Staleness Daemon

**Role:** Continuously audits the wiki for problems.

**Behavior:**
1. **Temporal contradictions:** For each entity, retrieve all claims sorted by `published_at`. Detect when a later claim contradicts an earlier one (use LLM to classify). Mark older as `superseded_by`. Surface delta on Talk page.
2. **Cross-perspective gaps:** For each major claim, check if both `bull` and `bear` sub-tenants have positions. If one is silent, it's a gap; surface as a Talk page note.
3. **Staleness:** For each entity, check most recent `ingested_at`. If older than threshold (30 days for fast-moving companies, 6 months for slow-moving labs), register a refresh function via HydraDB function routing.
4. **Unresolved Open Questions:** Aggregate across all entities; render the top-level "Open Questions" portal page.

### Stage 5: Function-Routed Self-Healing

**Role:** When the wiki goes stale, it heals itself.

**Behavior:**
- Each refresh function is registered as a knowledge object in HydraDB with metadata `entity_type: "function"`, `function_name: "refresh_figure"`, `function_url: "/api/refresh?entity=figure"`.
- When the staleness daemon detects a stale entity, it retrieves the registered function via recall and calls it.
- The function triggers a fresh ingestion pass for that entity.

**Demo note:** This is the most impressive feature for a HydraDB judge but the riskiest to build. Scope it down: implement function routing for exactly *one* entity (Figure), perfect that demo moment, leave the rest of the wiki on a manual refresh schedule.

**Modern-feeling parts where it's OK to break the Wikipedia aesthetic:**

- The Recent Changes live feed (can be a sidebar)
- The time-travel slider widget at the top of articles
- The entity path visualization (graph rendering)

### Page types to build

1. **Homepage** — Wikipedia-style portal. Featured Article of the Day, Categories grid, Recent Changes preview, "From the Archives" rotating section, Random Article button.

2. **Article page** (`/wiki/Figure_02`) — Infobox top right, prose body with footnoted citations, References section at bottom, Categories at very bottom. Tabs at top: Article | Talk | History.

3. **Talk page** (`/wiki/Talk:Figure_02`) — Threaded sections: Open Questions, Active debates (bull vs bear), Supersession log, Citation disputes.

4. **History page** (`/wiki/Figure_02?action=history`) — Wikipedia-style edit log. Each row: timestamp, editor agent name, edit summary, diff button. Clicking a past timestamp re-renders the article at that point in time.

5. **References / Footnote anchor pages** (`/wiki/Figure_02#cite_note-58`) — Standard Wikipedia footnote behavior.

6. **What Links Here** (`/wiki/Special:WhatLinksHere/Figure_02`) — List of every other article that references this entity, grouped by category.

7. **Disambiguation page** (`/wiki/Pi`) — "Pi may refer to: Pi (Physical Intelligence company), π (mathematical constant), pi0 (foundation model)."

8. **Recent Changes** (`/wiki/Special:RecentChanges`) — Live feed, descending. Each row: timestamp, article, editor agent, edit summary, diff link.

9. **Random Article** (`/wiki/Special:Random`) — Routes to a random article. Serendipitous browsing.

10. **Search results** (`/wiki/Special:Search?q=...`) — Search interface with semantic + lexical results from HydraDB recall.

11. **"Ask Embodipedia"** (overlay, Cmd+K) — Free-form natural language query that uses HydraDB recall + LLM synthesis. Results show source citations.

12. **Featured Article portal** — Quality-tiered article showcase. Demonstrates the evidence-density scoring.

## 15. Risk Register & Mitigations

### Risk 1: HydraDB rate limits during ingestion

**Likelihood:** Medium. **Impact:** High.

**Mitigation:**
- Batch knowledge memory writes (use bulk endpoint if available)
- Add retry logic with exponential backoff
- Cache extraction outputs locally before writing to HydraDB (so a write failure doesn't lose extraction work)
- Pre-warm by ingesting overnight before final demo day

### Risk 2: UI block runs long, demo lacks polish

**Likelihood:** High. **Impact:** Critical.

**Mitigation:**
- Hour 24 is a hard checkpoint. If Wikipedia visual language isn't done, cut killer feature #4 (self-healing).
- Have a fallback "lite UI" template ready that's still Wikipedia-styled but lacks fancy components
- Pre-write Wikipedia CSS as a separate stylesheet (hour 14, parallel to extraction work)

### Risk 3: Extraction quality is too low, articles look bad

**Likelihood:** Medium. **Impact:** High.

**Mitigation:**
- Manually review the first 20 extractions; iterate on the prompt
- Have a `manual_override.yaml` file for hand-corrected claims
- For demo entities (Figure, Optimus, π0), pre-write 10–15 hand-curated claims you know are accurate

### Risk 4: Time-travel slider doesn't render correctly

**Likelihood:** Medium. **Impact:** Medium (it's just one demo moment).

**Mitigation:**
- Build it as a server-side query, not client-side reconstruction
- Pre-cache 4 specific time snapshots (Jan 2024, Jan 2025, Jan 2026, Now) and let the slider snap to these instead of being continuous
- If broken at hour 46, cut from demo — keep the History tab without the slider

### Risk 5: Entity path visualization is buggy or ugly

**Likelihood:** Medium. **Impact:** Low (one demo moment).

**Mitigation:**
- Use a stable library (react-force-graph or cytoscape.js)
- Pre-cook one beautiful query result (Karol Hausman → Skild path) and hard-code that as the demo example
- Fall back to a textual chain ("A → B → C → D") if visualization breaks

### Risk 6: Function-routed self-healing is fragile

**Likelihood:** High. **Impact:** Medium.

**Mitigation:**
- Scope to one entity only (Figure)
- Pre-trigger the refresh just before recording demo so you can show it as a recorded interaction
- If it breaks live, skip moment 6 and replace with "look at Recent Changes — these are real edits from the agents that have been running"

### Risk 7: Judges don't grasp the perspective sub-tenant innovation

**Likelihood:** Low (HydraDB judges will get it). **Impact:** Medium.

**Mitigation:**
- The Talk page demo moment is engineered to make this self-evident
- Practice the line: "Three agents, same evidence pool, three different conclusions — each in their own HydraDB sub-tenant"
- Include the architecture diagram in the README so they can study it after

## 18. Appendix: HydraDB API Cheatsheet

(From docs.hydradb.com, current as of build start.)

### Tenant setup

```python
from hydradb import HydraClient

client = HydraClient(api_key=os.getenv("HYDRADB_API_KEY"))

# Create tenant (one-time)
client.create_tenant(
    tenant_id="embodipedia",
    metadata={
        "domain": "humanoid_robotics",
        "language": "en",
        "wiki_version": "1.0",
    }
)
```

### Writing a knowledge memory

```python
client.add_knowledge_memories(
    tenant_id="embodipedia",
    sub_tenant_id="canonical",   # or "bull" / "bear" / "articles"
    content="Figure 02 has been deployed at BMW since October 2024, performing logistics tasks.",
    metadata={
        "memory_id": "claim_figure_bmw_2024_10_15_001",
        "entity_type": "claim",
        "subject_entity": "Figure 02",
        "actor_entity": "Brett Adcock",
        "source_url": "https://twitter.com/adcock_brett/status/...",
        "source_type": "tweet",
        "evidence_strength": "primary",
        "published_at": "2024-10-15T14:32:00Z",
        "ingested_at": datetime.utcnow().isoformat(),
        "perspective": "neutral",
        "confidence": 0.72,
        "claim_type": "deployment",
    }
)
```

### Writing a hive memory (agent lesson)

```python
client.add_hive_memories(
    tenant_id="embodipedia",
    sub_tenant_id="agents",
    content="When @adcock_brett tweets a deployment timeline, historically the actual deployment slips 6 months on average. Discount confidence by 0.7x.",
    metadata={
        "lesson_type": "source_reliability",
        "subject": "brett_adcock_tweets",
        "confidence": 0.85,
    }
)
```

### Recalling from a specific sub-tenant

```python
results = client.full_recall(
    tenant_id="embodipedia",
    sub_tenant_id="canonical",
    query="Figure 02 deployment claims",
    metadata_filter={
        "subject_entity": "Figure 02",
        "claim_type": "deployment",
        "published_at": {"$gte": "2024-01-01"},
    },
    limit=20,
)
```

### Recalling hive memories

```python
lessons = client.memory_recall(
    tenant_id="embodipedia",
    sub_tenant_id="agents",
    query="reliability of brett adcock tweets",
    limit=5,
)
```

### Using entity_paths for "How is X connected to Y"

```python
results = client.full_recall(
    tenant_id="embodipedia",
    sub_tenant_id="canonical",
    query="connection between Karol Hausman and Skild",
)

# entity_paths returned with results
for path in results.entity_paths:
    print(path.summary)   # pre-computed multi-hop chain
    print(path.nodes)     # entities in the path
    print(path.edges)     # relationships
```

### Lexical recall for exact entity matches (infobox)

```python
# For infobox fields where you need exact matches
results = client.lexical_recall(
    tenant_id="embodipedia",
    sub_tenant_id="canonical",
    query="Figure 02 height",
    metadata_filter={"subject_entity": "Figure 02"},
)
```
